# METHOD.md — Полный workflow проекта

Методический документ для тебя как маркетолога-стратега. Описывает, как идёт проект от старта до сдачи. Для AI этот файл не предназначен — для AI есть `AGENTS.md`.

**Версия:** 2.0

---

## Принцип метода

```
Документ опережает код.
Сначала позиционирование → архитектура → бриф.
Только после утверждения каждого этапа — переход к следующему.
Код пишется по чёткому ТЗ, не импровизируется.
```

---

## Пять фаз проекта

```
ФАЗА 1 — Исследование и позиционирование
├── Outputs: market_research.md, target_audience.md, positioning_final.md
└── СТОП: позиционирование утверждено

ФАЗА 2 — Дизайн-система
├── Outputs: DESIGN_SYSTEM.md или выбран пресет
└── СТОП: дизайн-система утверждена

ФАЗА 3 — Архитектура сайта
├── Outputs: SITE_ARCHITECTURE.md + PAGE_*.md
└── СТОП: архитектура утверждена

ФАЗА 4 — Дизайн в Figma (опционально)
└── СТОП: макеты утверждены

ФАЗА 5 — Сборка проекта на Astro  ← здесь работает эта методика
└── СТОП: сайт собран, протестирован, задеплоен
```

## Workflow Фазы 5 (полный)

### Шаг 1. Подготовка папки проекта

```bash
# Создаёшь папку клиента
mkdir client-name && cd client-name

# Копируешь ядро методики
cp -r ~/metodika-astro-site/00_CORE/* .

# Копируешь шаблоны брифов
mkdir _briefs
cp ~/metodika-astro-site/05_BRIEFS_TEMPLATES/*.template.md _briefs/

# Инициализируешь git
git init
git add .
git commit -m "init: metodika astro site v2.0 starter"
```

### Шаг 2. Заполнение брифов

```bash
# Переименуй шаблоны (убери .template из имени)
cd _briefs
mv SITE_ARCHITECTURE.template.md SITE_ARCHITECTURE.md
mv PAGE.template.md PAGE_HOME.md
# и так далее

# Заполни каждый файл, опираясь на материалы Фаз 1-3
```

**Минимум для старта:** SITE_ARCHITECTURE.md + один PAGE_*.md (главной страницы)
**Максимум:** Все брифы заранее (Режим A — Greenfield)

### Шаг 3. AI-рекомендация stack

Открой папку проекта в любом AI-ассистенте. Отправь:

```
Прочитай AGENTS.md в корне проекта. В _briefs/ заполнены документы.
Прорекомендуй stack: дизайн-пресет, функциональные модули, анимации, header.
Покажи план файлов перед началом сборки.
```

AI:
1. Читает `AGENTS.md`
2. Анализирует `SITE_ARCHITECTURE.md` и `PAGE_*.md`
3. Извлекает нишу, размер проекта, особенности
4. **Рекомендует stack с обоснованием каждого выбора**
5. Ждёт твоего подтверждения

### Шаг 4. Подтверждение или корректировка

Получаешь от AI рекомендацию вида:
```
📐 Дизайн-пресет: Editorial Trust
🧩 Модули: marketing + blog + authors
🎬 Анимация: Standard
🧭 Header: Standard
```

Варианты ответа:
- «ок» → AI начинает сборку
- «поменяй на mega-menu» → AI корректирует и снова показывает
- «убери authors» → AI корректирует
- «давай Tech Forward вместо Editorial» → AI меняет пресет

Не торопись. Лучше потратить 2-3 минуты на корректировку рекомендации, чем переделывать собранный проект.

### Шаг 5. Сборка проекта

После твоего «ок» AI:
1. Копирует выбранные модули из библиотеки в проект клиента
2. Переносит брифы в финальные локации
3. Запускает `npm create astro@latest` для скаффолда
4. Устанавливает выбранные интеграции (Compressor, Sitemap, React, Tailwind)
5. Создаёт `src/` структуру по колокационной модели
6. Создаёт пустые блоки по числу секций в каждом `PAGE_*.md`
7. Импортирует блоки в роут-файлы

После этого можно запустить локально:
```bash
npm install
npm run dev
# → http://localhost:4321
```

### Шаг 6. Поблочное наполнение

Дальше по каждой странице:
```
Заполни блок 02 на главной — содержание из PAGE_HOME.md, секция "Проблема".
```

AI открывает один файл, наполняет, отчитывается. По очереди для всех блоков.

Это **самая длинная часть работы** — реальное наполнение контентом и стилизация под дизайн-систему. Закладывай 4-8 часов на проект 4-8 страниц.

### Шаг 7. GitHub

```bash
# Создаёшь приватный репозиторий на GitHub
# Имя по конвенции: client-[имя]-[год]
# Например: client-mosstroyresurs-2026

# Подключаешь локальный проект
git remote add origin git@github.com:user/client-mosstroyresurs-2026.git
git push -u origin main
```

### Шаг 8. Vercel

См. `VERCEL_GUIDE.md` для пошаговой инструкции.

Кратко:
1. На vercel.com → Add New Project
2. Выбираешь GitHub-репозиторий клиента
3. Vercel детектит Astro, подставляет команды
4. Жмёшь Deploy
5. Через 1-2 минуты сайт на client-name.vercel.app
6. Подключаешь домен клиента в Settings → Domains

### Шаг 9. Sitepins (если включён)

См. `CMS_GUIDE.md` для пошаговой инструкции.

Кратко:
1. На sitepins.com → подключаешь GitHub-репозиторий
2. Конфигурируешь поля редактирования
3. Получаешь admin URL для клиента
4. Тестируешь, что правки клиента триггерят деплой

### Шаг 10. Тестирование

**Обязательный набор тестов перед сдачей:**

- [ ] Lighthouse Mobile 90+, Desktop 95+
- [ ] Mobile вёрстка проверена на 375px (iPhone SE)
- [ ] Все CTA-кнопки кликабельны и ведут куда нужно
- [ ] Pop-up форма открывается со всех CTA
- [ ] Формы работают, заявки доходят до email
- [ ] Метрика подключена и считает события
- [ ] Sitepins работает (если активен): клиент может править контент
- [ ] Проверка через rich-results.google.com (JSON-LD валиден)
- [ ] Проверка скорости через PageSpeed Insights
- [ ] Проверка SEO через site:domain.ru в Яндекс/Google после деплоя

### Шаг 11. Сдача клиенту

Передаёшь клиенту пакет:
- Домен (логин/пароль регистратора если на твоём аккаунте, или передан клиенту)
- Vercel: либо передаёшь проект клиенту, либо он остаётся у тебя как managed
- Sitepins admin URL + обучение 15-30 минут (запись Loom видео для будущей справки)
- Доступ к Метрике
- Документация: что где править, к кому обращаться при проблемах

## Сопровождение клиентов после сдачи

Две модели в твоей системе:

### Модель А. С CMS (Sitepins активен)

- Клиент сам правит тексты, цены, контакты, добавляет статьи
- Ты подключаешься только для структурных изменений (новый блок, новая страница)
- Тариф: разовая оплата за сайт + опциональный консалтинг по запросу

### Модель Б. Без CMS (подписка на сопровождение)

- Все правки через тебя
- Ежемесячный тариф 5-15 тыс/мес покрывает мелкие правки
- Крупные изменения — отдельный счёт

**Какую модель выбирать с клиентом** обсуждай **до старта проекта**. От этого зависит цена и архитектура.

## Когда что-то идёт не так

| Проблема | Что делать |
|---|---|
| AI не может найти `AGENTS.md` | Проверь, что лежит в корне проекта, переоткрой сессию |
| Sitepins не показывает контент | См. `CMS_GUIDE.md` → раздел Troubleshooting |
| Vercel build падает | Открой Vercel → Deployments → Build Logs, AI пофиксит ошибку |
| Lighthouse <90 | Проверь Image optimization (только `<Image />`), убери GSAP если он не нужен |
| Path aliases не работают | Проверь `tsconfig.json`, baseUrl должен быть `.`  |
| Mobile menu не работает | Проверь, что компонент с `client:load`, не статичный |

## Куда расти со временем

После 5-10 проектов в новой системе:

1. **Накопи библиотеку паттернов** в `06_COMPONENT_PATTERNS/`. Это твой главный долгосрочный актив.
2. **Создай специализированные пресеты под ниши**: medical-trust, legal-warm, construction-industrial.
3. **Подними AI-промпты до собственного скилла** в Claude (если используешь Claude Code).
4. **Шаблоны брифов под ниши**: предзаполненные `SITE_ARCHITECTURE` под типовые B2B-сценарии.
5. **CI/CD**: GitHub Actions для автоматических Lighthouse-проверок при каждом push.
6. **Подрядчики**: когда поток вырастет, передавай рутину на подрядчика — твой `AGENTS.md` обеспечит консистентность.
