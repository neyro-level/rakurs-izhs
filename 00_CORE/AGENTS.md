# AGENTS.md

> **Назначение.** Универсальная инструкция для любого AI-ассистента, работающего с проектом по методике «Методика Астро сайт» v2.0. Совместимо с Claude Code, OpenAI Codex, Cursor, Kimi, Continue, Gemini.
>
> **Как использовать.** Файл лежит в корне репозитория с именем `AGENTS.md`. AI-ассистент читает его первым перед любой работой над проектом.
>
> **Альтернативные имена** для разных IDE: можно сделать симлинк `CLAUDE.md` (для Claude Code), `.cursor/rules/agents.md` (для Cursor), `AI_GUIDE.md` (общая конвенция). Источник истины один — этот файл.

## 1. Контекст проекта

Это многостраничный B2B-сайт на 4-8 страниц, опционально с блогом для SEO-лидогенерации.

**Стек проекта (фиксированный):**
```
Astro 6.2+              (требует Node 22+)
Tailwind CSS v4         (через @tailwindcss/vite)
TypeScript strict
React 18                только в островках
shadcn/ui               UI-компоненты в островках
Lucide React            иконки

Vercel                  хостинг
GitHub                  репозиторий
Sitepins                CMS (по умолчанию для всех проектов)
Яндекс.Метрика          аналитика
```

**Запрещено:** jQuery, Bootstrap, Vue, Angular, styled-components, CSS Modules, SPA-архитектура.

## 2. Source documents — что читать на старте

Перед любым кодом проверить три типа документов в `_briefs/` или в корне проекта:

| Документ | Где искать | Обязательность |
|---|---|---|
| `SITE_ARCHITECTURE.md` | `_briefs/` или корень | Обязательно |
| `DESIGN_SYSTEM.md` | `_briefs/` или корень | Обязательно (или используется пресет) |
| `PAGE_*.md` | `_briefs/` (потом переезжает в `src/pages/_pageName/`) | Минимум один |
| `BLOG_POST_*.md` | `_briefs/` (если блог-модуль активен) | Опционально |
| `AUTHORS.md` | `_briefs/` (если authors-модуль активен) | Опционально |

**Правило старта.**
1. Если `SITE_ARCHITECTURE.md` отсутствует — остановиться, запросить.
2. Если `DESIGN_SYSTEM.md` отсутствует — спросить, какой дизайн-пресет использовать (см. раздел 4).
3. Если ни одного `PAGE_*.md` нет — остановиться.

**Никогда** не выдумывать структуру страницы, тексты, цены, отзывы, кейсы, юр. формулировки. Если данных нет — пометить `<!-- TODO: replace with real content -->`.

## 3. Режимы работы

| Режим | Когда применяется | Что делает AI |
|---|---|---|
| **A. Greenfield** | Все брифы готовы заранее, нужен сайт за один проход | Рекомендация stack → подтверждение → полный скаффолд |
| **B. Incremental** | Готова главная и общая архитектура, остальные страницы добавятся позже | Скаффолд минимальной структуры с одной страницей, заглушки для остальных |
| **C. Maintenance** | Сайт уже собран, точечные правки или добавления | Идентификация → один файл → правка → отчёт |

Подробные стартовые промпты для каждого режима — в `CHEATSHEET.md`.

## 4. Module Recommendation Logic (главное)

### Принцип
При получении заполненного `SITE_ARCHITECTURE.md` AI **анализирует параметры проекта и рекомендует stack** до начала сборки. Не угадывает молча — явно показывает выбор и обоснование, ждёт подтверждения.

### Шаг 1. Извлечь параметры

Из `SITE_ARCHITECTURE.md` получить:
- Ниша клиента (стройка, ИЖС, юристы, бухгалтерия, агентства недвижимости, IT-агентство, производство, другое)
- Количество страниц
- Есть ли отраслевые подразделы (4+ страниц одного типа)
- Запрошен ли блог
- Целевая аудитория (B2B / B2C / mix)
- Тип бизнеса (премиум / средний / эконом)
- Особые пожелания клиента по визуалу

### Шаг 2. Подобрать модули

**Дизайн-пресет (выбрать один):**
- Editorial Trust → юристы, нотариусы, налоговые консультанты, адвокаты, медицинские эксперты
- Tech Forward → IT-агентства, SaaS, dev-shops, automation-сервисы, продуктовые tech-компании
- Chirkov B2B Premium (default) → ИЖС, стройка, ремонт, агентства недвижимости, производство, общий B2B-сервис, бухгалтерия среднего размера, B2B-агентства

Особые случаи:
- Клиент упомянул «как Linear/Stripe/Vercel» → Tech Forward
- Клиент упомянул «как Forbes/NYT» или «editorial» → Editorial Trust
- Клиент упомянул «как Apple» → Chirkov B2B Premium
- Если кастомный `DESIGN_SYSTEM.md` лежит в проекте — использовать его, не пресет

**Функциональные модули (добавить нужные):**
- `marketing-only` — всегда базовый, не модуль (включён по умолчанию)
- `blog` — добавить если в `SITE_ARCHITECTURE.md` упомянут блог
- `authors` — добавить ТОЛЬКО если есть блог И ниша = юристы/бух/мед/консалтинг (для E-E-A-T)
- `search-pagefind` — добавить если планируется 30+ статей в блоге

**Анимационный пресет (выбрать один):**
- `standard` — большинство классических B2B (стройка, юристы среднего сегмента, бухгалтерия)
- `premium` — премиум B2B, advisory, консалтинг с высоким чеком (от 500к за услугу)
- `wow` — только если клиент явно запросил презентационный/wow-эффект (агентства, design-shops)

**Header вариант:**
- `mega-menu` — если 4+ отраслевых страниц или 3+ услуг с подкатегориями
- `standard` — все остальные случаи

### Шаг 3. Сформировать рекомендацию

Вывести в ответе пользователю:

```

   [✓ blog — если применимо, обоснование]
   [✓ authors — если применимо, обоснование]
   [✓ search — если применимо, обоснование]

   Обоснование: [почему]

🧭 Header: [standard / mega-menu]
   Обоснование: [почему]

🚀 Деплой: Vercel (фиксировано в методике)
```

### Шаг 4. Дождаться подтверждения

**Не собирать проект до явного «ок» от пользователя.** Если пользователь корректирует выбор («поменяй на mega-menu» или «убери authors») — обновить план и снова показать перед началом.

### Шаг 5. После подтверждения

1. Скопировать выбранные модули из библиотеки `metodika-astro-site/` в проект клиента (или дать инструкцию пользователю это сделать)
2. Перенести `_briefs/` документы в финальные локации (`SITE_ARCHITECTURE.md` и `DESIGN_SYSTEM.md` в корень, `PAGE_*.md` в `src/pages/_pageName/`)
3. Скаффолдить Astro-проект через `npm create astro@latest`
4. Установить выбранные интеграции
5. Создать `src/` структуру по колокационной модели
6. Создать пустые блоки для каждой страницы по числу секций в `PAGE_*.md`
7. Импортировать блоки в роут-файлы

## 6. Path aliases (обязательно)

В `tsconfig.json` настроены aliases. **Все импорты только через них.** Относительные импорты `../` запрещены, кроме случаев когда файл в той же папке.

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"],
      "@layouts/*": ["src/layouts/*"],
      "@pages/*": ["src/pages/*"],
      "@lib/*": ["src/lib/*"],
      "@styles/*": ["src/styles/*"],
      "@assets/*": ["src/assets/*"],
      "@content/*": ["src/content/*"]
    }
  }
}
```

Использование:
```astro
---
import BaseLayout from '@layouts/BaseLayout.astro';
import { SITE } from '@lib/constants';
---
```

## 7. Поблочная модульность и протокол правок

### Принцип
Один блок = один файл = одна правка. Пользователь должен иметь возможность сказать «поправь блок 3 на главной» и получить изменения только в `src/pages/_home/03-Method.astro`.

### Жёсткие правила блоков

1. **Имена:** `NN-PascalCaseName.astro`, где `NN` — порядковый номер (`01`, `02`...)
2. **Один файл блока ≠ несколько секций.** Hero и Stats — это два файла, не один.
3. **Запрещены cross-block импорты внутри страницы.** Блок `02-Problem.astro` не должен импортировать `03-Method.astro`. Общая UI-часть выносится в `@components/ui/` или `@components/shared/`.
4. **Каждый блок самодостаточен.** Свой `<section>`, свои классы, свои копии.
5. **Стили локальные.** Tailwind utility-классы или `<style>` scoped к блоку.
6. **Копии в файле блока.** Заголовки, абзацы, CTA — в самом блоке.

### Протокол правок

1. Идентифицировать страницу. Если не указана — спросить.
2. Идентифицировать блок по номеру или имени из `PAGE_*.md`.
3. Открыть **ровно один файл** `src/pages/_pageName/NN-BlockName.astro`.
4. Внести изменение.
5. Не рефакторить, не переименовывать, не «улучшать» соседние блоки.
6. Отчитаться: «Изменён файл `src/pages/_home/02-Problem.astro`. Затронутые строки: NN-MM. Остальные блоки не тронуты.»

**Запрещённые паттерны при правках:**
- «Заодно почистил остальные блоки» — нет
- «Переименовал переменные ради консистентности» — нет
- «Вынес общую часть в shared, потому что увидел повторение» — спросить сначала
- Изменение глобальных стилей при правке одного блока — нет

## 8. Запреты — три уровня

### Уровень 1. Всегда запрещено (технически)

- `<h1>` в любом блоке кроме первого Hero (ломает SEO)
- Хардкод hex-цветов (только CSS-переменные из дизайн-системы)
- `!important` в CSS
- Inline-стили без обоснования
- Cross-block импорты внутри одной страницы
- React для статичных секций
- Произвольные правки соседних блоков
- Глобальные CSS-переопределения для одного блока
- Выдумывание отзывов, цифр, кейсов, имён, юр. формулировок
- Анимации, проседающие Lighthouse Mobile <90 / Desktop <95

### Уровень 2. По умолчанию избегать, но допустимо с обоснованием

- **Glassmorphism как декоративный приём** — точечно (один блок-выделение), не на всём сайте
- **Радиусы 16-24px** — для Tech Forward можно, для B2B Premium max 16px
- **Scale-hover** — допустим subtle (1.02-1.03), запрещён агрессивный (1.05+)
- **Тёмные зоны** — только для ритма, не «просто для красоты»
- **Gradient в текстах** — допустим в Hero H1 одной страницы, не везде
- **Параллакс** — на 1-2 элементах, не везде

### Уровень 3. Опционально для премиум-проектов (только при выборе пресета premium/wow)

- GSAP scroll-triggered анимации
- Framer Motion в React-островках
- Lenis smooth scroll
- Animated counters
- Stagger animations
- Animated text reveal

### Glass-эффект navbar — отдельно

**Backdrop-blur на sticky header — разрешён и рекомендуется.** Это технический паттерн для читаемости sticky-навигации (используется на apple.com, stripe.com, linear.app, vercel.com). Не путать с запрещённым glassmorphism как декоративным приёмом по всему сайту.

## 9. React-островки — когда можно

`client:load` или `client:visible` только для:
- Mobile menu toggle
- FAQ accordion (если не нативный `<details>`)
- Контактная форма с валидацией
- Калькулятор / фильтр / табы со state
- Модальное окно

**Запрещено** для статичных карточек, секций, иконок.

## 10. SEO, доступность, производительность

**SEO:**
- Уникальные `<title>` (max 60) и `<meta description>` (max 160) на каждой странице
- JSON-LD `Organization` на главной, `FAQPage` в FAQ-секциях, `Article`/`BlogPosting` на статьях
- `robots.txt` и sitemap (через @astrojs/sitemap) обязательно
- Canonical URL на каждой странице
- Open Graph: динамическая генерация og-image для блога

**Доступность:**
- Связанные `<label>` для инпутов
- Контраст не ниже WCAG AA (4.5:1)
- Hover **и** focus на всех интерактивных элементах
- Семантические теги (`<nav>`, `<main>`, `<section>`, `<article>`, `<footer>`)

**Производительность:**
- Lighthouse Mobile 90+, Desktop 95+ (target)
- Изображения только через `<Image />` из `astro:assets`
- WebP/AVIF форматы
- Hero-изображение: `fetchpriority="high"`, без `loading="lazy"`
- Astro Compressor (gzip/brotli) подключён в `astro.config.mjs`

## 11. Стандарты кода

- TypeScript `strict: true`. `any` запрещён без комментария-обоснования.
- `const` / `let`, никогда `var`
- Никаких `console.log` в production
- `<button>` для действий, `<a>` для навигации, не путать
- Один `<h1>` на страницу — только в Hero (блок 01)
- Иерархия заголовков h1 → h2 → h3 без пропусков
- Все `<img>`: `width`, `height`, `alt`, `loading`
- Все формы: связанные `<label>`
