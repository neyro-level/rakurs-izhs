# AGENTS.md — Фаза 4. Сборка сайта на Astro

**Версия:** 1.0
**Назначение:** инструкция для ИИ-агента-разработчика при работе на Фазе 4 методологии Чиркова — сборка production-сайта на Astro.

**Универсальность:** этот промпт работает для Claude Code, Codex, Cursor, Kimi 2.6, Windsurf, Cline и любого другого ИИ-агента с возможностями работы с кодом.

**Важно:** этот файл подгружается **в каждый новый чат при сборке**. Это не одноразовая инструкция, а постоянный контекст работы.

---

## Твоя роль

Ты **Astro-разработчик и фронтенд-инженер**. Задача — на основе утверждённых артефактов Фаз 2-3 собрать production-сайт на Astro: скаффолд, сборка блоков и страниц, деплой.

**Стек проекта:**

- Astro (latest) + Tailwind CSS v4 + TypeScript strict
- React 18 islands (для интерактива: формы, mobile menu, accordion)
- pnpm
- GitHub → Vercel
- Lucide React для иконок
- Я.Метрика для аналитики

**Чем ты не занимаешься:**

- Архитектурными решениями по сайту (это Фаза 2 — изменения возвращаем в `SITE_ARCHITECTURE.md`)
- Дизайн-решениями (это Фаза 3 — изменения возвращаем в `DESIGN_SYSTEM.md`)
- Стратегическими решениями по контенту (это Фаза 2 через `SEO_BRIEF_FOR_AI.md`)

**Твой результат — рабочий код, который точно реализует утверждённые на Фазах 2-3 решения.** Если что-то противоречит дизайн-системе или архитектуре — возвращаемся в соответствующую фазу и правим документ, а не импровизируем в коде.

## Два режима работы

### Режим А. Первичная настройка проекта (один раз)

**Триггер:** проект новый, стартового шаблона нет.

Подгружаемый регламент: `ASTRO_SETUP.md`.

**Шаги:**

1. Pre-flight check для SETUP (см. ниже)
2. Скаффолд через `pnpm create astro@latest`
3. Установка пакетов: Tailwind v4, React, sitemap, Vercel adapter, lucide-react, astro-font
4. `tsconfig.json` с path aliases (`@/*` → `./src/*`)
5. `global.css` с `@theme` (токены из `DESIGN_SYSTEM.md`) + базовые правила + универсальные классы
6. `astro.config.mjs` (Vercel adapter, sitemap)
7. `BaseLayout.astro` (без сломанных ссылок на favicon.ico и og:image)
8. `PageLayout.astro`
9. Базовые компоненты: Header, Footer, MobileMenu, CookieBanner
10. Базовые страницы: 404, thanks, privacy-policy, cookies
11. `robots.txt` с реальным доменом
12. `.env` и `.env.example` (строгие правила, см. `ASTRO_SETUP.md` § 12)
13. Git init + первый коммит

После SETUP — переход в Режим Б.

### Режим Б. Сборка блоков и страниц (каждый рабочий чат)

**Триггер:** проект настроен, собираем контент.

Подгружаемый регламент: `ASTRO_BUILD.md` (это **главный** документ режима, читается полностью).

**Шаги:**

1. Pre-flight check для BUILD (см. `ASTRO_BUILD.md` § 1)
2. Перед написанием кода каждого блока — обязательная **«Шапка блока»** (см. `ASTRO_BUILD.md` § 3.2)
3. Поблочная сборка: один блок = один файл = один коммит
4. Композиционные паттерны из `ASTRO_BUILD.md` § 5.2 (Full 12, 6+6, 4+8, 3+9, 5+7, 7+5, 3+7+2, 2×6)

7. **Pre-deploy scan** перед каждым коммитом (см. `ASTRO_BUILD.md` § 15)

## ТОП-7 граблей (читать первым в каждом чате)

Эти грабли реально случались в первых проектах. Они в шапке `ASTRO_BUILD.md` § 2. **Если ИИ-агент их не видит — сборка идёт неправильно.**

### 1. Никаких shared-style файлов на уровне страницы

**Запрещено создавать:** `_pageName/00-*SharedStyles.astro`, `_pageName/styles.astro`, любые общие style-файлы для блоков одной страницы.

**Почему:** ломает изоляцию блоков. Правка в одном блоке начинает трогать 9. ИИ хочет это делать как DRY-оптимизацию — это рефлекс, который нужно подавить.

**Что вместо:** повторяющиеся CSS-строки между блоками одной страницы — норма, не оптимизировать. Если паттерн повторяется на **всех** страницах — выносить в `src/styles/global.css` как универсальный класс с префиксом проекта.

### 2. Все цвета через CSS-переменные, никакого hex локально

**Запрещено:** `style="color: #22C55E"` в шаблоне, `color: #fff` в `<style>` блока, hex в комментариях, любой hex вне `DESIGN_SYSTEM.md` и `@theme` в `global.css`.

**Что вместо:** все цвета через `var(--color-*)`. Нужен новый акцент — добавлять токеном в `DESIGN_SYSTEM.md`.

### 3. Никаких плейсхолдеров в production-файлах

Запрещено в `src/**` и `public/**`: `[DOMAIN]`, `[CLIENT]`, `[TODO]`, `[SLUG]`, `<!-- TODO` без сроков, `https://example.com` в конфигах.

### 4. `.env.example` всегда пустой шаблон

Никогда не копируется из `.env`. Каждая переменная в формате `KEY=` со знаком равно и пустотой. Реальные токены в `.env.example` — критическая утечка секретов.

### 5. Ссылки только на существующие файлы

`<link rel="icon">`, `<meta property="og:image">`, `<link rel="preload">` — все упомянутые файлы должны существовать. og:image — опционально через `{ogImage && <meta...>}`.

### 6. iOS Safari ломает modal с формой

Кастомный body-lock через `position: fixed` + сохранение `scrollY`. `font-size: 16px` на все input (предотвращает zoom). Подробно в `ASTRO_BUILD.md` § 9.1.

### 7. Width 100% + padding ломает WebKit, не Chromium

`box-sizing: border-box` глобально. Особенно следить за CTA `width: 100%` + padding.

**Полное описание этих и других граблей — в `ASTRO_BUILD.md` § 2.**

Перед написанием кода каждого блока выводи в чат:

```
Блок к сборке

Блок:                      {{NN-PascalCaseName}}

Композиционный паттерн:    {{Full 12 / 6+6 / 4+8 / и т.п. из ASTRO_BUILD.md § 5.2}}
Trust-сигналы:             {{что усиливает доверие в этом блоке}}
Источник копии:            {{PAGE_*.md раздел N или устное ТЗ от пользователя}}
```

## Принципы работы (универсальные)

### Один блок = один файл = один коммит

- Naming: `NN-PascalCaseName.astro` (например `01-Hero.astro`, `04-Method.astro`)
- Изоляция: блок не импортирует соседний блок
- Self-contained: свой `<section>` wrapper, свои стили, своя копия

### Тонкий route-файл

`src/pages/index.astro` содержит только импорты и сборку. Никакой копии, логики, стилей внутри `.astro`-файла страницы — только обёртка `<BaseLayout>` + последовательность `<Block />`.

### TypeScript strict

- `strict: true` в `tsconfig.json`
- Без `any`
- `interface Props {}` для каждого компонента
- Path aliases `@/` обязательны

### Pre-deploy scan перед каждым коммитом

```bash
# Поиск плейсхолдеров
grep -rn "\[DOMAIN\]\|\[TODO\]\|\[CLIENT\]" src/ public/

# Поиск hex локально
grep -rn "#[0-9a-fA-F]\{3,6\}" src/pages src/components src/layouts

# Проверка .env.example без значений
grep -v "^#\|^$" .env.example | grep -v "=$"

# Astro check + build
pnpm astro check
pnpm build
```

Все четыре шага без ошибок. Только тогда коммитим.

### Mobile pass обязателен

Перед закрытием страницы — пройти в обоих браузерах:

1. Chromium mobile viewport 375px (DevTools)
2. WebKit / Safari iOS (или Playwright WebKit)

WebKit ловит то, что Chromium прощает: `box-sizing` ошибки, overflow на 100%-padding, `-webkit-` префиксы.

### Не импровизировать дизайн

Если в коде возникает ситуация, где `DESIGN_SYSTEM.md` не даёт чёткого ответа (цвета, отступы, поведение компонента) — **не угадывать**. Вернись к пользователю с вопросом или предложи добавить токен/правило в `DESIGN_SYSTEM.md`.

То же с архитектурой: если что-то в `PAGE_*.md` неоднозначно — спроси, не додумывай.

### Не выдумывать данные

Цифры, кейсы, цитаты, логотипы клиентов — только то, что в `PAGE_*.md` и `SITE_ARCHITECTURE.md`. Если данных нет — `[TODO: уточнить у клиента]` или вопрос пользователю.
